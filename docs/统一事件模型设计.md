# 统一事件模型设计

## 摘要

本文档定义 PRD v2.1 在 P0 阶段使用的统一事件模型，用于连接控制面与执行面。目标是让 Dashboard、策略闸门、A2A、Knowledge Flywheel、Detector/Fixer 都基于同一套事件契约读写与追踪，避免每个模块各自定义日志与状态。

本设计遵循以下前提：

- v2.0 继续作为当前主干方案
- v2.1 按 P0 / P1 分期落地
- 当前仓库几乎没有 v2.1 直接实现基础
- 控制面与执行面必须分离

---

## 目标

统一事件模型需要解决四个问题：

1. 让所有核心行为有统一、结构化、可检索的记录方式
2. 让策略判断、审批、执行、回退都可追踪
3. 让多租户数据天然隔离
4. 为后续 Dashboard、知识沉淀、自动化修复提供稳定输入

---

## 设计原则

### 一、事件优先于功能

在 v2.1 中，Dashboard、Hooks、Flywheel、A2A 都依赖统一事件。没有统一事件，就不应该先做这些上层能力。

### 二、事实来源与投影分离

- **执行面** 负责产出事实事件
- **控制面** 负责消费事件并生成判断、审批、策略结果
- **Dashboard** 只是事件投影，不是事实来源

### 三、默认租户隔离

所有事件必须带 `tenant_id`，所有消费与查询都必须先过租户边界。

### 四、可审计、可回放

每个关键事件必须可关联上下游链路，支持回溯与问题复盘。

### 五、向后兼容

事件版本要允许逐步演化，不能一改字段就破坏所有消费者。

### 六、统一入口先归一，再入事件流

统一事件模型不仅是存储格式，还必须是所有入口的归一目标。

P0 阶段要求：

- cron 触发先转换为标准任务事件
- webhook / 外部回调先转换为标准任务事件
- 人工操作先转换为标准任务事件
- A2A 请求先转换为标准任务事件

统一入口后的最小链路应保持一致：

`task.created -> task.classified -> policy.evaluated -> approval / execution`

如果不同入口绕开统一事件链，后续就会出现权限判断、审计与回放不一致的问题。

---

## 通用事件结构

所有事件统一使用以下结构：

```json
{
  "event_id": "evt_xxx",
  "event_type": "finding.created",
  "schema_version": "1.0.0",
  "tenant_id": "tenant_xxx",
  "source": "detector.monitor",
  "created_at": "2026-03-28T14:00:00Z",
  "correlation_id": "corr_xxx",
  "causation_id": "evt_parent_xxx",
  "trace_id": "trace_xxx",
  "status": "open",
  "qaps_class": "S",
  "autonomy_level": "manual",
  "decision": "require_approval",
  "artifact_refs": [],
  "metadata": {},
  "payload": {}
}
```

---

## 通用字段定义

### 必填字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `event_id` | string | 全局唯一事件 ID |
| `event_type` | string | 事件类型 |
| `schema_version` | string | 事件结构版本 |
| `tenant_id` | string | 租户 ID |
| `source` | string | 事件来源 |
| `created_at` | string | UTC 时间戳 |
| `status` | string | 当前状态快照 |
| `payload` | object | 事件体 |

### 推荐字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `correlation_id` | string | 同一业务链路关联 ID |
| `causation_id` | string | 触发本事件的上游事件 ID |
| `trace_id` | string | 链路追踪 ID |
| `qaps_class` | string | QAPS 分类结果 |
| `autonomy_level` | string | 自治级别 |
| `decision` | string | 策略输出 |
| `artifact_refs` | string[] | 证据、日志、报告等引用 |
| `metadata` | object | 扩展元数据 |

## 事件类型清单

P0 阶段至少定义以下事件：

### 任务类

- `task.created`
- `task.classified`
- `task.closed`

### 策略类

- `policy.evaluated`
- `approval.requested`
- `approval.resolved`

### 问题与修复类

- `finding.created`
- `finding.updated`
- `fix.proposed`
- `fix.executed`
- `fix.failed`
- `fix.rolled_back`

### Agent 协同类

- `a2a.requested`
- `a2a.executed`
- `a2a.expired`

### Hooks / 学习类

- `hook.triggered`
- `knowledge.candidate_created`
- `knowledge.promoted`

### 入口归一类

- `ingress.received`
- `ingress.normalized`
- `ingress.rejected`

## 状态机建议

### Task 状态

`created -> classified -> policy_evaluated -> awaiting_action -> completed | cancelled | failed`

### Finding 状态

`open -> acknowledged -> confirmed -> in_progress -> resolved | false_positive`

### Proposal 状态

`proposed -> dry_run_passed -> pending_approval -> approved | rejected -> executing -> executed | failed -> rolled_back`

### Approval 状态

`requested -> pending -> approved | rejected | expired`

### A2A 状态

`requested -> dispatched -> replied -> completed | expired | blocked`

要求：

- 状态转换必须由事件驱动
- 不允许静默跳状态
- 所有异常路径都必须留下审计记录

---

## 存储与索引建议

### 主索引

- `event_id`
- `tenant_id + created_at`
- `correlation_id`
- `event_type`

### 推荐查询维度

- 按租户查询最近事件
- 按链路回溯事件树
- 按 finding / proposal / task 查询全生命周期
- 按 risk_level / qaps_class 统计

### 存储职责

- 事件总线：负责传输
- 事件仓库：负责查询与投影
- 审计仓库：负责保留不可变记录

不建议让 Dashboard 直接读执行库事实表作为唯一来源。

---

## 校验规则

### 入库校验

1. `tenant_id` 不能为空
2. `event_type` 必须属于白名单
3. `created_at` 必须是合法 UTC 时间戳
4. `payload` 必须符合对应事件 schema
5. 高风险事件必须带 `correlation_id`

### 消费校验

1. 同一 `event_id` 重复到达时必须幂等
2. 无 schema_version 的事件默认拒收
3. 无 tenant_id 的事件不得进入共享流
4. 关键事件缺少 artifact 引用时要标记为审计不完整

---

## 审计要求

所有以下事件必须同时写入审计：

- `policy.evaluated`
- `approval.requested`
- `approval.resolved`
- `fix.executed`
- `fix.failed`
- `fix.rolled_back`
- `a2a.requested`
- `a2a.executed`

审计记录至少包含：

- actor
- tenant_id
- event_id
- 前后状态
- 触发原因
- 证据引用
- 时间戳

此外，所有入口归一动作也必须留下审计：

- 原始入口类型（cron / webhook / manual / a2a）
- 原始载荷引用
- 归一后的任务 ID
- 归一失败原因（如被拒绝）

---

## 租户隔离要求

### 强制要求

- 所有事件都带 `tenant_id`
- 跨租户查询默认禁止
- 知识候选事件默认不共享
- A2A 事件必须在租户边界内显式校验

### 不允许的做法

- 用全局共享事件表但无 tenant 过滤
- 在 Flywheel 中默认聚合所有租户案例
- 用通用 root 消息绕过租户检查

---

## 回退与 kill switch

事件模型本身需要支持回退能力，因此必须保留：

- `fix.rolled_back`
- `approval.expired`
- `a2a.expired`
- `status` 明确回退结果

同时要求控制面支持：

- 全局 kill switch
- 租户级 kill switch
- 单任务 kill switch

---

## P0 验证标准

P0 阶段至少验证以下项目：

1. 核心事件可正常生成与消费
2. Dashboard 能读取统一事件投影
3. Detector/Fixer 能沿统一事件流完成一条最小链路
4. A2A 能产生结构化事件且可追溯
5. 审计库中能按 `correlation_id` 还原一次完整链路
6. 不同入口进入的任务都能被归一到同一事件链

---

## P1 演进方向

P1 可以在不破坏本模型的前提下增加：

- 更细粒度的 Hook 事件
- Knowledge Flywheel 的评分与晋升事件
- 更复杂的策略链路事件
- Dashboard 写操作事件

前提是：继续保持 schema 向后兼容，不破坏 P0 消费者。
