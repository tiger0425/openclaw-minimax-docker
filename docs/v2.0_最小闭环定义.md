# v2.0 最小闭环定义

## 摘要

本文件定义 v2.0 的第一个“最小可运行闭环”。目标不是一次性做成完整体系，而是先交付一个**真实可运行、可验收、可回退**的版本。

当前约束已经很明确：

- v2.0 还没有实施
- v2.1 不能整体并入当前周期
- 允许并行的只有：**v2.0 主线 + v2.1 最小基础预埋**

因此，v2.0 的首个闭环必须小、清楚、可复现，并且不依赖完整 v2.1 控制面能力。

---

## 目标

v2.0 最小闭环的目标只有三个：

1. **把一条真实任务链路跑通**
2. **把一份真实交付物产出来**
3. **把这条链路的状态、日志和回退留清楚**

只要这三件事能稳定完成，v2.0 的第一阶段就算成立。

---

## 闭环定义

一句话定义：

> **需求输入 → Noah 调度 → 执行 Agent 产出 → 基础质检 → 交付物输出 → 状态与日志留存**

这就是 v2.0 最小闭环。

它强调的是：

- 真正有输入
- 真正有人/系统调度
- 真正有执行
- 真正有输出
- 真正可追踪

而不是只有概念、只有文档、或者只有局部原型。

---

## 必须上线的角色

v2.0 第一阶段必须上线的不是全部 00-07 角色，而是最小闭环所需角色。

## 1. 任务入口

负责接收需求或任务指令。

职责：

- 接收任务
- 生成任务标识
- 写入最小状态记录
- 把任务交给 Noah

形式可以是：

- CLI
- API
- 固定输入文件

当前阶段不要求多入口并存，但必须保证以后能扩展。

## 2. Noah（最小调度器）

这是 v2.0 主干里最关键的角色。

职责：

- 接收任务
- 判断交给哪个执行 Agent
- 记录调度状态
- 汇总交付结果

当前阶段的 Noah 只需要具备“最小调度能力”，不需要具备完整控制面能力。

## 3. 至少一个执行 Agent

第一阶段不要求 02-07 全部都产品化上线，但必须至少有一个真正可执行工作的 Agent。

建议优先：

- 04 号执行/实施能力

原因是它最接近“能产出真实交付物”的主线角色。

## 4. 基础质检环节

不要求完整 06 号体系，但必须有一个基础验收动作，确认交付物是否成功生成。

可以是：

- 文件存在检查
- 结果格式检查
- 基础运行成功检查

## 5. 交付输出位

必须有明确的交付物存放位置与命名规则。

不要求完整客户门户，但至少要做到：

- 交付物可定位
- 可访问
- 可用于后续验收

---

## 暂不上线的能力

以下内容不进入 v2.0 最小闭环：

- 完整 7 Hooks 体系
- Knowledge Flywheel
- Detector / Fixer 自动闭环
- Dashboard 控制写操作
- 完整 A2A 协议
- 完整策略闸门
- 完整多租户治理

这些都属于下一阶段增强，不是 v2.0 第一个闭环成立的必要条件。

---

## 最小流程

## 步骤一：任务进入

输入一个真实任务，例如：

- 一个客户需求
- 一个待执行任务说明
- 一个需要产出交付物的请求

要求：

- 能被系统接收
- 能生成唯一标识
- 能留下最小记录

## 步骤二：Noah 调度

Noah 负责：

- 接收任务
- 指定执行 Agent
- 记录当前状态

此处最小要求不是复杂调度策略，而是“能把任务明确地下发出去”。

## 步骤三：执行 Agent 工作

执行 Agent 接收任务后，必须：

- 真正执行任务
- 真正产出结果
- 把结果回传或落盘

没有真实产出，就不算闭环。

## 步骤四：基础质检

系统需要做一个最小的结果确认，例如：

- 交付物是否存在
- 格式是否正确
- 是否达到了最基本成功条件

## 步骤五：交付物输出

产物必须进入一个固定位置，例如：

- `deliverables/`
- 指定输出目录
- 指定存储卷

## 步骤六：状态收口

最终必须能查到：

- 成功还是失败
- 产物在哪
- 谁执行的
- 花了多久
- 出错时错在哪

---

## 最小交付物

v2.0 最小闭环必须至少产出以下内容：

1. **一份真实交付物**
2. **一条完整任务记录**
3. **一份最小验收结果**
4. **一份部署与运行说明**
5. **一份回退说明**

这里的“真实交付物”可以先不追求花哨，但必须是业务上有意义的结果，而不是内部占位文件。

---

## 最小运行环境

当前仓库最现实的选择，是优先利用现有部署骨架。

建议最小运行环境包括：

- `openclaw-minimax-docker/` 作为主运行目录
- 一套可启动的容器编排或等价运行脚本
- `.env.example` 中的必要环境变量
- 本地卷或固定目录用于存放日志与交付物

当前阶段不假设：

- 外部复杂控制面
- 完整监控平台
- 独立事件总线
- 完整数据库治理体系

---

## 最小日志与状态要求

每个任务至少要能记录以下字段：

- `task_id`
- `correlation_id`
- `source`
- `timestamp`
- `state`
- `executor`
- `outcome`
- `artifact_location`
- `error_message`
- `duration_ms`

这些字段的作用不是为了做完整 v2.1，而是为了让 v2.0 第一版：

- 能追踪
- 能排错
- 能验收
- 能回退

---

## 与 v2.1 的兼容边界

在 v2.0 最小闭环里，允许保留的 v2.1 兼容内容只有两类：

## 1. 统一入口归一层的接口位

要求：

- 主流程不要和某个单一入口强耦合
- 预留标准任务上下文结构
- 不要求现在实现完整入口治理

## 2. 两层执行安全的接口位

要求：

- 保留应用层的可判断边界
- 保留宿主 / 工具层的最终拦截边界
- 不要求现在实现完整审批和策略链

不允许：

- 让 v2.0 主线依赖 v2.1 控制面才能运行
- 在当前周期启用完整控制逻辑

---

## 验证标准

v2.0 最小闭环成立，至少要满足以下条件：

1. 能提交一个真实任务
2. Noah 能接收并下发任务
3. 至少一个执行 Agent 能真正执行并产出结果
4. 系统能做最小质检
5. 交付物有固定位置可查
6. 任务状态和关键日志字段可查
7. 有最小部署说明和回退说明

如果上述任一项缺失，就不能称为“v2.0 最小闭环已成立”。

---

## 不通过条件

以下任一情况出现，v2.0 第一阶段都不视为通过：

1. 任务只能输入，不能真正被执行
2. Noah 只是概念存在，没有实际调度动作
3. 执行 Agent 没有真实产出
4. 交付物无法定位或无法复用
5. 没有日志或状态记录，导致无法排错
6. 无法说明如何启动、如何验收、如何回退
7. 提前引入完整 v2.1 控制面，导致主线复杂度失控

---

## 下一步建议

在这份《v2.0 最小闭环定义》之后，最合理的下一份文档应该是：

- `docs/v2.0_第一阶段开发任务清单.md`

因为现在闭环边界已经确定，下一步就应该进入：

- 具体拆任务
- 明确先后顺序
- 明确哪些任务属于主线，哪些属于副线预埋

如果继续推进，建议顺序如下：

1. 先写《v2.0_第一阶段开发任务清单》
2. 再确认主运行目录中的实际改造点
3. 然后才开始进入代码实施

---

## 最终口径

v2.0 的首个“完成”不是“功能都齐了”，而是：

> **这条最小链路已经能真实跑通、真实产出、真实验收、真实回退。**

做到这一点，v2.0 才算真正开始成立；之后再谈 v2.1 的增强能力，才不会失控。
