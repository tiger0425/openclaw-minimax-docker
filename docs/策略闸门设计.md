# 策略闸门设计

## 摘要

本文档定义 PRD v2.1 在 P0 阶段的策略闸门体系。其目标是把 `QAPS`、`Autonomy Ladder`、`A2A 两步触发` 三套原本容易分散的判断逻辑，统一收口为一条策略链，让系统在“是否执行、是否审批、允许谁执行、允许调用什么”上保持一致。

策略闸门属于**控制面核心模块**，它只负责判断与授权，不直接修改执行事实。

---

## 设计目标

策略闸门需要解决以下问题：

1. 让所有任务先分类，再决定执行边界
2. 让授权与审批不再分散在多个模块里重复实现
3. 让 A2A、Fixer、Hooks 都服从同一条策略链
4. 让所有高风险行为都可审计、可回退

---

## 核心原则

### 一、单链路决策

所有关键动作必须走同一条判断链：

`任务进入 -> 分类 -> 风险评估 -> 自治级别 -> 是否审批 -> 是否允许执行 -> 审计记录`

### 二、分类与授权分离

- `QAPS` 只回答“这是什么类型的任务”
- `Autonomy Ladder` 只回答“这个任务允许谁、以什么方式执行”

它们不能互相替代。

### 三、A2A 服从策略，不自带权限体系

A2A 只是触发机制，不是授权来源。所有跨 Agent 行为必须先经过策略闸门判断。

### 四、默认保守

P0 阶段所有高风险动作默认：

- `require_approval`
- `manual`
- `dry-run`

---

## 输入与输出

在进入策略闸门前，所有请求必须先经过统一入口归一层，不允许 cron、webhook、人工入口、A2A 分别带着各自的隐式上下文直接进入判断逻辑。

### 输入

策略闸门接收统一评估请求，建议结构如下：

```json
{
  "request_id": "req_xxx",
  "tenant_id": "tenant_xxx",
  "actor": {
    "id": "noah",
    "type": "agent"
  },
  "entity": {
    "type": "task",
    "id": "task_xxx"
  },
  "requested_action": "execute_fix",
  "context": {
    "qaps_class": "S",
    "artifacts": [],
    "history_event_refs": []
  }
}
```

### 输出

```json
{
  "decision": "require_approval",
  "autonomy_level": "manual",
  "risk_level": "high",
  "requires_approval": true,
  "allowed_tools": ["read", "write"],
  "allowed_agents": ["builder"],
  "rules_hit": [
    {
      "rule_id": "rule_high_risk_change",
      "reason": "系统级变更"
    }
  ],
  "ttl_seconds": 600
}
```

## 策略判断链

### 第一步：任务分类（QAPS）

在任务分类之前，必须已经完成入口归一。策略闸门只消费标准化后的任务上下文，不直接消费原始入口载荷。

将任务归为：

- `Q`：一次性问答
- `A`：小型交付物
- `P`：多步骤项目
- `S`：系统级变更

分类结果进入后续策略判断，不直接决定执行权限。

### 第二步：风险评估

风险建议统一分为：

- `low`
- `medium`
- `high`
- `critical`

风险评估要综合：

- 是否写状态
- 是否跨租户
- 是否影响生产链路
- 是否涉及删除 / 发布 / 对外发送
- 是否涉及 A2A 多轮协作

### 第三步：自治级别（Autonomy Ladder）

建议统一为：

- `manual`：只能人工执行
- `suggest`：Agent 只能提出建议
- `auto-low`：低风险自动执行
- `auto-high`：白名单场景自动执行

P0 阶段原则：

- `high / critical` 默认不得进入 `auto-*`
- `S` 类任务默认不得自动执行

### 第四步：审批判定

以下情况强制 `require_approval`：

- `QAPS = S`
- 风险等级为 `high` 或 `critical`
- 涉及删除、发布、对外发送
- 涉及跨 Agent 派单并伴随写操作
- 涉及策略修改、知识晋升、租户共享

### 第五步：工具与 Agent 白名单

策略输出必须明确：

- 允许调用哪些工具
- 允许触达哪些 Agent
- 是否允许 A2A
- 是否允许继续 spawn 子 Agent

这样才能避免同一个任务在不同入口下得到不同权限。

### 第六步：两层执行安全对齐

策略闸门输出不是最终执行本身，而是第一层安全边界。P0 必须明确两层执行安全：

#### 第一层：控制面安全

- 判断是否允许执行
- 判断是否需要审批
- 输出工具白名单
- 输出 Agent 白名单
- 输出 TTL 与执行范围

#### 第二层：宿主 / 工具层安全

- 对危险命令做最终阻断
- 对越权工具调用做最终拒绝
- 对高风险写操作保留强制 fail-closed 拦截

原则是：

- prompt 不是最终安全边界
- 策略允许不等于宿主必须放行
- 高风险执行至少保留一层最终硬约束

## 规则模型

建议规则以结构化方式管理，而不是散落在代码里。

### 规则字段

| 字段 | 说明 |
|---|---|
| `rule_id` | 规则唯一 ID |
| `name` | 规则名称 |
| `priority` | 优先级 |
| `tenant_scope` | 生效租户范围 |
| `conditions` | 命中条件 |
| `effect` | 规则效果 |
| `enabled` | 是否启用 |
| `version` | 版本号 |

### 规则效果示例

- 强制审批
- 禁止执行
- 降级为建议模式
- 限制工具白名单
- 限制 Agent 白名单
- 限制 TTL 与最大轮次

---

## 与 A2A 的关系

### A2A 两步触发必须走策略链

A2A 建议流程：

1. 生成 root message
2. 发起 `a2a.requested`
3. 调用策略闸门评估
4. 若允许，则实际派发
5. 若需审批，则进入审批流

### A2A 必须继承的约束

- 最大往返轮次
- TTL
- 子 Agent 禁止继续派单
- 来源与目标权限矩阵
- 审计记录

不允许 A2A 绕过策略链直接执行。

---

## 与 Dashboard 的关系

Dashboard 只能展示策略输出与审批状态，不能自己代替策略引擎做判断。

P0 中 Dashboard 允许：

- 查看最近策略决策
- 查看命中规则
- 查看审批状态

P0 中 Dashboard 不允许：

- 跳过审批
- 临时改策略后直接执行
- 手工篡改决策结果

---

## 审计要求

每次策略评估都必须产生 `policy.evaluated` 事件，并记录：

- 请求来源
- 输入上下文
- 命中规则
- 风险等级
- 自治级别
- 最终决策
- 过期时间（TTL）

所有审批动作必须有：

- 审批人
- 审批时间
- 审批理由
- 关联任务 / finding / proposal

---

## 决策失效与回退

### TTL 机制

所有决策都应有有效期。超时后若仍未执行，必须重新评估。

### 冲突处理

如果两个并发决策冲突：

- 优先使用更高优先级规则
- 若仍冲突，默认采用更保守结果
- 记录冲突审计事件

### kill switch

必须支持：

- 全局自动化关闭
- 租户级自动化关闭
- 指定任务自动化关闭

kill switch 触发后，所有 `auto-*` 自动退化为 `manual` 或 `suggest`。

### 统一入口失败处理

如果入口归一失败，策略闸门不得继续评估，而应：

- 直接返回拒绝
- 生成 `ingress.rejected` 审计记录
- 明确给出失败原因

---

## P0 验证标准

1. QAPS、Autonomy、A2A 已接入同一策略链
2. 高风险动作全部进入审批流
3. 所有策略输出可在 Dashboard 中查询
4. 所有策略输出可在审计库中追溯
5. kill switch 生效后，自动执行立即收敛
6. 不同入口不会因上下文差异而得到冲突决策
7. 宿主 / 工具层对越权执行仍能做最终阻断

---

## P1 演进方向

P1 可以继续增加：

- 租户自定义规则包
- 更细粒度工具权限
- 基于历史成功率调整自治级别
- 低风险自动执行放量

前提是 P0 已经证明：

- 决策链稳定
- 审计链完整
- 审批与回退可用
