# v2.0 / v2.1 双轨同步实施方案

## 摘要

本方案给出一种可执行的“双轨同步”策略：在保证可控、可回退的前提下，允许 v2.0 主线落地与 v2.1 的最小基础预埋并行推进。明确原则是：只允许 **v2.0 主线 + 最小 v2.1 基础** 并行，不允许完整 v2.1 控制面或自动化能力在 v2.0 未稳定前并入。

这里的“最小 v2.1 基础”仅指两项：

- **统一入口归一层**
- **两层执行安全**

目标是最小化返工、保证可观测与审计，并为后续完整 v2.1 平滑切换铺路。

---

## 结论

### 可以同步，但不能全量同步

可以并行推进的，是：

- **v2.0 的主干实现**
- **v2.1 的最小基础预埋**

不能并行推进的，是：

- **完整 v2.1 控制面**
- **完整自动化能力**
- **完整自我进化能力**

因此，当前正确的同步方式不是“2.0 和 2.1 一起做完”，而是：

> **主线做 v2.0，可运行、可交付、可验收；副线做 v2.1 基础骨架，但不打开高级能力。**

---

## 为什么不能全量同步

### 一、v2.0 还没有实施

当前最核心的现实约束是：**v2.0 还是方案，不是现成系统。**

这意味着：

- 你们还没有一个已经稳定运行的主链路
- 也没有一个现成底座可以承接完整 v2.1
- 如果此时把完整 v2.1 一起拉进来，就会同时改业务主链路和控制治理链路

### 二、v2.1 的复杂度属于控制面复杂度

v2.1 不是几个附加功能，而是一套新的控制面能力，包括：

- Hooks
- 路由学习
- Detector / Fixer
- Dashboard
- A2A
- 审批与策略链

如果在 v2.0 尚未跑通时同步实现这些能力，风险会集中出现在三个方面：

1. **难验收**：很难判断是 v2.0 主链路有问题，还是 v2.1 控制面引入了问题
2. **难回退**：控制面和执行面一起改，出问题时回退范围会过大
3. **难调试**：主线逻辑和治理逻辑交叉，排错效率会很低

### 三、返工风险会被放大

如果现在直接把完整 v2.1 和 v2.0 一起做：

- 要么把当前实现做得过重
- 要么未来仍然要为了治理和审计重构

所以最优做法不是全量同步，而是**分层同步**。

---

## 哪些可以同步

以下内容适合同步推进：

### 1. v2.0 主干功能

这是双轨中的主线，必须优先推进：

- 需求输入
- Noah 调度
- 执行 Agent 输出
- 基础质检
- 基础交付链路
- 部署与运行说明

### 2. v2.1 的统一入口归一层

这是未来最容易造成返工的结构点之一，因此可以在 v2.0 中先预埋。

当前阶段只需做到：

- 入口适配与业务处理解耦
- 定义统一任务上下文
- 主流程不直接绑定某种单一入口格式

### 3. v2.1 的两层执行安全

这也是适合同步预埋的基础能力。

当前阶段只需做到：

- 保留控制层的可判断边界
- 保留宿主 / 工具层的最终阻断边界
- 留出未来接入审批、白名单、策略的接口位置

### 4. 最小审计与最小日志

虽然完整控制面不应提前实现，但最小日志与审计字段可以同步前置，例如：

- `task_id`
- `correlation_id`
- `source`
- `timestamp`
- `outcome`

这些不会显著抬高当前复杂度，但能为后续升级省掉大量返工。

---

## 哪些不能同步

以下能力不应与 v2.0 主线一起在当前周期完整实施：

### 1. 完整 7 Hooks 体系

包括：

- `before_prompt_build`
- `before_tool_call`
- `after_tool_call`
- `agent_end`

等全链路机制，不应在当前周期整体上线。

### 2. Knowledge Flywheel

候选知识、评分、晋升、注入、回写，这一整套能力都应后置。

### 3. Detector / Fixer 自动闭环

可以先保留契约思路，但不应在 v2.0 周期里开放完整自动修复链路。

### 4. Dashboard 控制能力

可读的基础监控可以保留，但不应引入写控制、审批动作、运维操作入口。

### 5. 完整 A2A 协议

当前阶段可以保留接口思路，但不应在主线未稳定前把 A2A 扩成正式复杂协议。

### 6. 完整策略闸门系统

当前阶段只应预埋执行边界，不应实现完整的控制面决策链。

---

## 主线与副线定义

## 主线：v2.0 主干落地

主线目标是：

- 让 v2.0 真实可运行
- 让交付流程真实跑通
- 让系统具备可部署、可验证、可交付的最小能力

主线关注的是“业务主链路能不能成立”。

## 副线：v2.1 最小基础预埋

副线目标是：

- 避免未来返工
- 固定关键边界
- 预留后续演进接口

副线不追求“功能上线”，只追求“结构不被未来推翻”。

---

## 双轨阶段划分

## 阶段一：范围收敛与路线冻结

### 主线任务

- 定义 v2.0 最小闭环
- 确定第一阶段上线职责
- 确定第一批交付物

### 副线任务

- 确定统一入口归一层的最小接口
- 确定两层执行安全的最小边界
- 确定需要提前保留的字段与拦截点

### 交付物

- v2.0 最小闭环定义
- 双轨实施边界说明
- 入口归一接口草案
- 执行安全接口草案

---

## 阶段二：v2.0 主链路实现 + v2.1 最小预埋

### 主线任务

- 建立基础运行环境
- 打通需求 → Noah → 执行 Agent → 交付物输出主链路
- 建基础日志与错误捕获

### 副线任务

- 在入口层预留统一入口适配位
- 在执行层预留最终拦截位
- 增加最小 trace / 审计字段

### 交付物

- 可运行主链路
- 基础日志方案
- 入口适配占位实现
- 执行安全占位实现

---

## 阶段三：v2.0 稳定化 + v2.1 基础验证

### 主线任务

- 完成端到端验收
- 完成部署文档、使用说明、维护说明
- 明确回退方案

### 副线任务

- 验证统一入口归一层不会破坏主线
- 验证两层执行安全接口可用
- 验证最小审计字段可追溯

### 交付物

- v2.0 验收记录
- 回退说明
- 接口验证结果
- 最小审计验证结果

---

## 阶段四：进入完整 v2.1 的切换评估

### 主线任务

- 判断 v2.0 是否已稳定可交付

### 副线任务

- 评估是否进入 v2.1 P0
- 决定是否开始统一事件模型工程化
- 决定是否开始策略闸门与只读 Dashboard

### 交付物

- v2.1 P0 准入评审结论
- 后续阶段实施建议

---

## 每阶段交付物

| 阶段 | 主线交付物 | 副线交付物 |
|---|---|---|
| 阶段一 | v2.0 最小闭环定义、职责裁剪版 | 双轨边界说明、接口草案 |
| 阶段二 | 可运行主链路、基础日志方案 | 入口适配占位、执行安全占位 |
| 阶段三 | 验收记录、部署说明、回退说明 | 接口验证结果、最小审计验证 |
| 阶段四 | v2.0 稳定结论 | v2.1 P0 准入评审 |

---

## 风险与控制点

### 风险一：副线越做越重，拖慢主线

#### 控制点

- 副线必须只做基础接口与最小占位
- 不允许把完整控制面逻辑提前做进去
- 所有副线能力默认 feature flag 关闭

### 风险二：完全没有预埋，未来返工

#### 控制点

- 统一入口归一层必须留接口
- 两层执行安全必须留边界
- 最小 trace / 审计字段必须前置

### 风险三：主线与副线互相污染

#### 控制点

- 主线代码和副线占位要尽量分层
- 副线不应改写主线业务判断逻辑
- 所有副线变更都必须说明“不影响主线运行”

### 风险四：过早把原型机制产品化

#### 控制点

- 不允许当前周期上线完整 Hooks
- 不允许当前周期上线自动修复闭环
- 不允许当前周期上线知识自动回写

---

## 进入完整 v2.1 的切换条件

只有满足以下条件，才建议从“双轨同步”切换到“正式实施完整 v2.1 P0 / P1”：

### 条件一：v2.0 主链路已稳定

- 端到端链路真实可跑
- 交付物可产出
- 部署说明可执行
- 回退方案可用

### 条件二：基础日志与审计已具备

- 关键任务可追溯
- 最小 trace 字段可查
- 错误与结果可记录

### 条件三：入口归一与执行安全接口已被验证

- 统一入口接口可工作
- 宿主 / 工具层拦截点可工作
- 不会破坏当前主线运行

### 条件四：团队已准备好承接控制面复杂度

- 愿意接受更高治理成本
- 愿意接受分阶段灰度与审计要求
- 愿意对 hooks / 策略 / 自动化能力做持续维护

如果以上条件不满足，就不应进入完整 v2.1，而应继续稳住 v2.0 主线。

---

## 最终建议

最终建议是：

- **允许双轨同步，但必须是“主线做 v2.0，副线只做 v2.1 最小基础”**
- **不允许把完整 v2.1 和 v2.0 一起并行交付**
- **先把 v2.0 做成真实可交付系统，再切换进入完整 v2.1**

换句话说：

> 可以同步，但只能分层同步；不能功能级全量同步。

这才是当前既稳、又不容易返工的实施方式。
